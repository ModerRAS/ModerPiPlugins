# ModerPiPlugins 开发约定

## 项目定位

- 本仓库用于存放个人自用的 pi 插件/扩展相关代码与配置。
- 以小而清晰为原则，不预设复杂 monorepo 结构。
- 在没有明确要求时，不要为了“统一”而重排目录。

## 参考项目

- `ref/` 目录用于存放参考项目、上游实现、临时对照代码。
- `ref/` 是只读参考区，默认不提交到 Git。
- 除非用户明确要求，不要修改、清理、移动或提交 `ref/` 下的内容。
- 需要借鉴实现时，优先做最小化迁移，不要整段照搬参考项目的大块结构。

## 修改原则

- 优先做最小必要改动，避免顺手重构无关内容。
- 修改前先读取相关文件，不要只根据文件名猜测结构。
- 未经用户确认，不要删除看起来是有意保留的代码、配置或目录。
- 如果要引入依赖、脚本或新的工程子目录，先保证该目录自包含且用途明确。

## 目录与文件约定

- 新增插件或实验内容时，保持单目录自包含，至少包含：
  - 源码
  - 最小使用说明
  - 必要配置文件
- 公共说明放在仓库根目录 `README.md`。
- 针对单个插件的使用方法、限制、示例，放在各自目录的 `README.md`。

## 文档要求

- 新增插件、删除插件或明显变更用法时，同步更新对应文档。
- 文档以直接、可执行为准，避免空泛描述。
- 如果某个实现依赖 pi 的特定机制，写清前置条件、入口方式和示例。

## pi 相关约定

- 如果需要参考 pi 上游实现或写法，可查看 `ref/pi/`。
- `ref/pi/` 仅用于对照，不代表当前仓库必须跟随其目录结构、命名或工程规范。
- 引用上游行为时，优先保留当前仓库的轻量结构，只迁移当前任务真正需要的部分。

## Git 约定

- 只提交本次任务实际修改的文件。
- 不要使用会覆盖他人或历史工作的破坏性命令。
- 明确排除 `ref/`，不要把参考项目混入正式提交。

## 命令执行

- 运行命令前先确认仓库里实际存在对应脚本或工具。
- 当前仓库比较轻量；如果没有 `package.json`、测试脚本或构建脚本，不要臆造固定工作流。
- 如果后续新增某个插件自己的构建/测试命令，应优先在该插件目录内定义和执行。
